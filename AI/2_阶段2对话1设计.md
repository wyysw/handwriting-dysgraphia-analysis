## 三、关键步骤：从 maze_mask 构造通道几何（这是阶段2最核心的新工作）

这一步决定了后续四个 F 特征的质量，所以需要仔细设计。基本思路文档已经给出，但我想把操作步骤明确化：

**步骤 1 —— 外框填充得到画布内部区域 `canvas_mask`**  
`maze_mask.png` 是一张前景=墙壁线条的二值图。迷宫最外层一圈墙是封闭的矩形框。

- 方法：先做一次轻微膨胀（2–3 px）确保外框连通，然后用 `cv2.floodFill` 从画布四角以外的"背景像素"出发，把外部填满；取反即可得到"外框内部区域"`canvas_mask`。
- 备选方案（更稳健）：如果 floodFill 因线条不闭合而失败，可退化为"取墙壁 mask 的最大外接矩形 → 矩形内部填 1"。建议两种方法都实现，主方法失败自动回退。

**步骤 2 —— 通道掩码 `channel_mask`**  
```
channel_mask = canvas_mask AND NOT dilate(wall_mask, r_wall)
```
`r_wall` 约 2–3 像素，用于确保墙壁两侧通道**干净分离**（不因墙的细缝连通）。

**步骤 3 —— 通道骨架 `channel_skeleton`**  
`skimage.morphology.skeletonize(channel_mask > 0)`。对方形迷宫，骨架基本是一条由直线段组成的折线；但若迷宫含死胡同/分叉，骨架会有分支。这里**不做分支剪枝**，保留完整骨架作为参考结构（F2 采样也基于它）。

**步骤 4 —— 通道距离变换 `channel_dist`**  
`channel_dist = distanceTransform(1 - channel_mask, DIST_L2)` —— 每个非通道像素到最近通道的距离，用于 F3 的越界加权。

## 四、F1–F4 的具体实现

| 特征 | 公式 | 关键参数 |
|---|---|---|
| **F1 通道覆盖率**（↑） | `F1 = |user_mask ∩ channel_mask| / |channel_mask|` | — |
| **F2 骨架采样点命中率**（↑） | 沿骨架每 `sample_step` 像素取一个采样点，计命中点比例（与用户 mask 距离 ≤ `hit_radius`） | `sample_step=20`, `hit_radius=8` |
| **F3 无效书写加权距离**（↓） | `F3 = Σ channel_dist[user_mask ∩ ¬channel_mask] / |channel_mask|` | — |
| **F4 通道外笔迹比**（↓） | `F4 = |user_mask ∩ ¬channel_mask| / |user_mask|` | — |

几处细节我想明确说一下：

**关于 F1 语义**。文档写的是"用户笔迹落在通道内的像素数 / 通道总像素数"，即 **recall 语义**。隐含假设是"正确的走法应该贯穿整条通道"。这对走迷宫类任务基本成立（迷宫整体就是一条单路径或主路径加短岔道）。如果实际迷宫有大段无效岔道，F1 会系统性偏低——这时应考虑改为"用户笔迹到通道的 precision"或 IoU。**建议先按文档实现 recall，观察真实分布后再决定是否调整。**

**关于 F2 骨架采样的顺序**。简单按像素坐标行优先遍历骨架点再每 20 个取一个是可行的，但对评估"走到多远"这种线索更直觉的做法是**沿骨架弧长等距采样**：
1. 调用 `cv2.findContours` 或自己做骨架遍历拿到有序路径点；
2. 计算累积弧长；
3. 每 `sample_step` 弧长取一点。  
弧长采样法也能直接迁移到圆形迷宫（按弧度即按弧长）。

**关于 F3 是否需要"越界惩罚"**。对称游戏里有"越轴惩罚"，因为对称轴是个硬约束。迷宫没有对称轴，但是否有"画出画布外框"或"穿越墙壁"的强惩罚？我的建议是：F3 里使用 `channel_dist`（违规像素到最近通道的距离）自然体现了"错得多远就罚多重"——穿墙一格和画到纸外的惩罚力度自动拉开。**不额外加硬惩罚系数**，保持实现简单。

## 五、C1/C2/C3（直接复用公共库）

这三个指标的公式在迷宫任务上完全不变，**唯一差异是 C1 的参考结构**：对称游戏用 Hough 线段，迷宫用 `channel_skeleton`。

**公共库里 C1 的接口设计**我建议做成：
```
compute_C1_jitter(strokes, reference, jitter_tol, channel_half_width) -> float
  reference 可以是:
    (a) List[(a, b)] —— 直线段列表（对称游戏用）
    (b) np.ndarray   —— 骨架二值图 / 距离变换图（迷宫用）
```
内部逻辑统一为"点到参考结构的垂直（最近）距离"：
- 对 (a)：点到每条线段的垂足投影距离；
- 对 (b)：直接查 `distanceTransform(~skeleton)` 的值。

`channel_half_width` 的作用与对称游戏相同——过远的点视作"根本没在参考结构附近"，从统计中剔除以避免空笔飘远造成的伪抖动。

**C2 在迷宫上的一个潜在问题**：阶段1的文档里指出 C2 = 短笔段总长/所有笔段总长，阈值是"画布对角线 × 0.02"。但结合刚才观察到的 "l3/l4 全程一笔完成"，**迷宫样本的笔画数可能只有 1–2 个**，C2 要么恒为 0（没有短笔段），要么极端（唯一笔段低于阈值）。这会让 C2 在迷宫上几乎丧失判别力。

两种处理方式：
1. 照常计算，接受 C2 在迷宫上失效；让阶段 3 的归一化 + 阶段 4 的分类器自己学习其权重（可能会接近 0）。
2. 把 C2 在迷宫上改造为"笔画总数 / 理论最少笔画数（= 1）"之类的替代指标。

**我倾向方式 1**，因为文档明确要求"三游戏统一"，保持方法论一致比压榨每个指标的性能更重要。这一点需要你确认。

**C3 压力变异系数**：对应到迷宫数据，`analyze.py` 的 `split_into_strokes_simple` 会丢弃 pressure=0 的点，笔段内部的压力全 > 0，可以直接套阶段1的公式。

## 一、核心架构调整

你的(3)和(6)两点把设计改变最大。原计划里"通道几何 = 参考结构"这一条被拆成了两层：

```
maze_mask.png
  │
  ▼  ① 通道几何构造（与之前相同）
channel_mask, canvas_mask, channel_dist
  │
  ▼  ② 【新增】solve_maze_path 子模块 —— 提取最快通关路径
entry_xy, exit_xy, 
solution_polyline,          # 沿弧长有序的点列
solution_skeleton_mask,     # 1px 宽的解路径骨架
solution_channel_mask       # 解路径周边的"正确通道"(给 F1 用)
  │
  ├──► F1 = user ∩ solution_channel / solution_channel   （分母是解路径通道，不是全通道）
  ├──► F2 = 沿 solution_polyline 等弧长采样的命中率
  ├──► F3 = 用 channel_mask(全通道) 做 valid_zone，加权距离
  └──► F4 = user 落在 channel_mask(全通道) 外的比例
```

关键区分：**F1/F2 用"解路径"为基准，F3/F4 用"全通道"为基准**。这符合直觉：
- 功能完成度（F1/F2）要求"走对了"——只看最短解；
- 越界/偏离（F3/F4）惩罚"走出通道"——走错岔路但仍在通道内不算越界。

这样设计的附加好处是：走错岔路（走进死胡同又退回）会让 F1 偏低（没覆盖到解路径上的某些段）和 F4 正常（仍在通道里），形成一个可区分的信号组合，这比"通道整体=路径"的简化模型信息量更大。

## 二、回答你的(3)：channel_skeleton 不等于最快通关路径

`channel_mask` 的骨架化会把**所有可行走区域的中轴线**都提取出来，包括迷宫的死胡同和环路分支。看你上传的 maze_mask.png，这个迷宫的通道明显不是单条路径，有大量死胡同（目测 15+ 处），骨架化后会产生一棵"树+少量环"的结构，入口到出口的解路径只是这棵树上的一条简单路径。

所以**需要新增一个子模块**。下面详细描述它的实现（这是阶段2最核心的新工作）。

## 三、新子模块：`solve_maze_path(channel_mask, wall_mask)`

**步骤 1：自动检测入口 / 出口**

```
canvas_bbox = bbox_from_mask(wall_mask)           # 外框包围矩形
frame_ring = 四条边上的像素集合 (宽 1-2 px)
openings = frame_ring ∩ channel_mask               # 框上"可走"的像素
connected_components(openings) → 应得到 2 个连通域
  → 每个连通域取其质心作为入口/出口锚点
```

对你这张 maze_mask.png，入口在右上、出口在左下——两个连通域位置明确，直接返回 2 个 `(x, y)` 锚点。若自动检测不到恰好 2 个，fallback 到"用户在函数参数里手动指定 entry_xy、exit_xy"。

**步骤 2：基于骨架建图，BFS/Dijkstra 求最短路径**

```
skeleton = skimage.morphology.skeletonize(channel_mask)
entry_anchor = 骨架上距离入口锚点最近的像素
exit_anchor  = 同上
构造图 G：
  节点 = 骨架像素
  边   = 8 邻接，权重 = 欧氏距离 (直=1, 对角=√2)
path_pixels = shortest_path(G, entry_anchor, exit_anchor)
             （用 scipy.sparse.csgraph.dijkstra 或手写 BFS 都行）
```

骨架像素约 10³ 量级，图算法瞬间完成。对这张迷宫我预估解路径长度在 3000–5000 像素。

**步骤 3：生成三种输出表示**

```
solution_polyline       : path_pixels 去抖化后的有序 (x,y) 列表（可选 Douglas–Peucker 简化）
solution_skeleton_mask  : path_pixels 渲染成 1px mask，用于可视化调试
solution_channel_mask   : dilate(solution_skeleton_mask, r = 估算的通道半宽)
                          用于 F1 的分母
```

`r` 的估算：对 `channel_dist` 在 `channel_skeleton` 上取值，取中位数即为通道半宽的估计值。这样 `solution_channel_mask` 的形态跟真实通道的对应段基本吻合。

## 四、F1/F2 的具体公式（更新后）

**F1 通道覆盖率 (解路径)**：
```
F1 = |user_mask ∩ solution_channel_mask| / |solution_channel_mask|
```
解读："用户笔迹覆盖了解路径多大比例"。走错岔路不会提升 F1（因为岔道不在 `solution_channel_mask` 里），走得远但沿着解路径会提升 F1。

**F2 解路径采样点命中率**：
```
沿 solution_polyline 按弧长每 sample_step 像素取一个点 → sample_pts (K 个)
dist_user = distanceTransform(~user_mask)
hit = Σ 1[dist_user[p] <= hit_radius]
F2 = hit / K
```
沿弧长采样有个好处：**一次实现就能迁移到圆形迷宫**（圆形时换成按弧长等距采样，本质是按弧度等距）。

参数建议：`sample_step = 40 px`（对 1201×1601 画布约 30 个采样点），`hit_radius = 12 px`（略宽于笔迹线宽，允许轻微偏离）。实际跑出来再调。

## 五、C1 的设计：用户自导出的霍夫线段（采纳你的(6)）

这确实是一个比"到参考骨架的距离"优雅得多的设计。关键洞察：**C 轴测量的是"运动控制能力"**（能不能把一条直线画直），跟"画的是不是对的路径"是两个正交维度。所以 C1 的参考不必来自 target，来自用户自己的"应画的意图"即可。

对你给的这张方形迷宫特别合适——除转角处以外用户的笔迹理想情况下都是**横向或纵向直线段**，Hough 能很稳定地把这些"用户应该画成直线"的部分识别出来。

**实现：把 `sym_feature_extractor.compute_C1_jitter_ratio` 直接复用，只换 `hough_segments` 的来源。**

```
user_mask = render_trajectory_using_reference_bbox(txt, png, canvas_hw, line_thickness=3)
user_hough_segments = extract_segments_from_hough(user_mask)   # 复用 sym_analyze3._extract_target_segments_from_hough
C1, bad, total = compute_C1_jitter_ratio(
    mapped_strokes, user_hough_segments,
    jitter_tol=3.0, channel_half_width=channel_half_width,
)
```

`_extract_target_segments_from_hough` 在 sym_core 里现成，它本身已经包含了按角度分类（h/v/diag_pos/diag_neg）→ 聚合共线片段 → 去除过短段的完整流程。**唯一可能需要调整的是 HoughLinesP 的阈值**：

| 参数 | sym 目标图形 | 用户笔迹(方形迷宫) | 理由 |
|---|---|---|---|
| `threshold` | 25 | 20–25 | 用户笔迹有抖动，投票数略低 |
| `minLineLength` | 25 | 40–60 | 方形迷宫的直线段较长，排除短过渡段 |
| `maxLineGap` | 6 | 10 | 用户笔迹有小间隙 |

**建议**：先直接复用 sym 的默认参数跑一次 l3/l4，观察 Hough 输出是否合理，再决定是否需要为"用户笔迹"单独开一组阈值。`extract_segments_from_hough` 可以接一个可选的 `hough_params` 字典，方便调参。

**边界情况 —— Hough 找不到足够线段时**：
- 正常样本：应有几十条合并后的线段；
- 极端抖动的障碍样本：可能只有少数几条或零条。

这种情况下"投影命中数 total"会很小，C1 = bad/total 不稳定。我建议在 meta 里额外记录 `projected_fraction = total / num_user_points`，作为"线段一致性程度"的辅助诊断。**核心公式与 sym 完全一致（`C1 = bad / projected`）**，以满足"三游戏统一"的方法论约束。

## 九、需要最后确认的两个小点

1. **入口出口自动检测失败的 fallback**：是让函数抛异常要求用户手动传 `entry_xy`/`exit_xy`，还是静默使用一个默认启发式（例如"最靠近 bbox 四角的通道像素"）？我倾向前者——迷宫样本数量不大，异常时手动标注比冒错提取解路径的风险小。

2. **solution_channel_mask 的"粗化半径" r**：是动态估算（从 `channel_dist` 中位数）还是固定值（比如 20 像素）？动态估算更稳，但如果通道宽度在整个迷宫里基本一致，固定值也完全够用。看你上传的迷宫，通道宽度基本恒定，固定 r 可能更可控。我倾向先用**动态估算的中位数**，记录在 meta 里，后续发现不稳再改固定。
